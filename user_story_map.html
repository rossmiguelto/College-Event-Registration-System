CERS Part 5 — User Story Map & Setup Guide

College Event Registration System · Ross Miguel Luna & Arash Nail · Red Deer Polytechnic

1. User Description

Primary User: College Student (Registrant)

A full-time or part-time student at Red Deer Polytechnic, aged 18–28, comfortable with smartphones and web browsers. They want a quick, frustration-free way to sign up for campus events without paperwork. They check the dashboard occasionally to confirm their registration went through.

Goals: Browse events → register in under 2 minutes → see confirmation.

Pain Points: Long paper forms, no confirmation receipt, not knowing if they got a spot.

Secondary User: Department Administrator (Analyst)

A college staff member responsible for event planning, aged 25–50. They need to monitor registration counts, spot the most popular events, and track day-to-day engagement without exporting spreadsheets manually.

Goals: See live registration stats → identify popular events → ask data questions quickly.

Pain Points: Manually counting registrations, no real-time overview, digging through raw database rows.

2. User Story Map

BACKBONE — User Journey
1. Discover Events 2. Register 3. Confirm 4. Monitor (Admin) 5. Analyse (Admin)
As a student, I want to browse upcoming events on the home page so I can decide which one to attend.
As a student, I want to search events by keyword so I find what I care about fast.
As a student, I want to fill in my name and email on a form so the system knows who I am.
As a student, I want to select an event from a dropdown so I don't mistype the event name.
As a student, I want the form to tell me if I forgot a required field before submitting.
As a student, I want to see an alert confirming my registration was saved to the database.
As a student, I want to be redirected to the dashboard after registering so I can see my entry.
As an admin, I want to see a table of all registrations with names, events, and dates so I have a live audit trail.
As an admin, I want KPI cards showing totals so I get the key numbers at a glance.
As an admin, I want a bar chart of registrations by event so I can see which events are most popular.
As an admin, I want a line chart of daily registrations so I can see engagement trends over time.
As an admin, I want to filter the bar chart by minimum count so I can hide low-traffic events.
As an admin, I want to ask a chatbot plain-English questions and get live database answers.
Walking Skeleton (MVP): Home page → Form → SQLite save → Dashboard table
Release 1: + KPI cards + bar chart + line chart (Flask/Python)
Release 2: + Chart filter dropdown (JS amends visual) + chatbot tool

3. Instructions to Run

Prerequisites

  • Python 3.10 or newer — python.org
  • Visual Studio Code with the Live Server extension (optional — Flask can serve everything)

Step-by-step

  1. Open the project folder in VS Code: File → Open Folder → CERS_Part5
  2. Open a new terminal: Terminal → New Terminal
  3. Install Python packages: 
    pip install -r requirements.txt
  4. Start Flask: 
    python app.py
    You should see: 
    ✅ Database initialized.
🚀 Flask server starting at http://localhost:5000
  5. Open your browser and navigate to: 
    http://localhost:5000
    The full site is served by Flask — no Live Server needed.
  6. To test the registration form: go to Register for Events, fill in the form, and click Submit Registration. You will be redirected to the Dashboard where your entry appears in the table and the charts refresh automatically.
  7. To test the chatbot: on the Dashboard, type a question like "How many registrations are there?" and press Enter or click Ask.

HTTP Status Codes from Terminal (Rubric Item 6b)

After submitting a registration and loading the dashboard, your terminal will show lines like:

127.0.0.1 - - [08/Apr/2026 10:24:01] "POST /api/register HTTP/1.1" 201 -
127.0.0.1 - - [08/Apr/2026 10:24:02] "GET /dashboard.html HTTP/1.1" 200 -
127.0.0.1 - - [08/Apr/2026 10:24:02] "GET /api/dashboard HTTP/1.1" 200 -
127.0.0.1 - - [08/Apr/2026 10:24:02] "GET /api/stats HTTP/1.1" 200 -
127.0.0.1 - - [08/Apr/2026 10:24:03] "GET /api/charts/events HTTP/1.1" 200 -
127.0.0.1 - - [08/Apr/2026 10:24:03] "GET /api/charts/timeline HTTP/1.1" 200 -

200 OK — The request succeeded; the server returned the requested resource (HTML page, JSON data, or PNG chart image).

201 Created — The POST /api/register request succeeded and a new row was inserted into the database. 201 is more precise than 200 for creation operations.

400 Bad Request — Returned if the form submission is missing required fields (username, email, or event_name). Tells the client to fix its request.

500 Internal Server Error — Returned if the database connection fails or an unhandled exception occurs server-side. Indicates a server-side problem, not a client problem.
